## 2. Critical findings

Each entry: location ¬∑ what ¬∑ why ¬∑ sketch ¬∑ severity.

### C1 ‚Äî `skipRest()` leaves `timerSeconds` non-zero before next exercise [KNOWN ‚Äî PATCH IN FLIGHT]

[app.ts:695](app.ts#L695). `skipRest()` clears the interval and `state.isResting=false` but does NOT reset `state.timerSeconds`. The next render path that reads `state.timerSeconds` for the timed-exercise button label ([app.ts:1276](app.ts#L1276)) will see a leftover value and either hide the "Start timer" button under `disabled` or display the rest-timer's leftover number as the wall-sit duration. **Fix sketch:** in `skipRest()` set `state.timerSeconds = 0`. Severity: **CRITICAL** (loses ability to start wall-sit timer until quit/restart).

### C2 ‚Äî Pull-merge on boot can resurrect deleted-locally or stale records, never updates by id

[app.ts:849-903](app.ts#L849). `pullFromSupabase` filters remote rows by `!localIds.has(r.id)` and prepends; if a local row has same id as remote it is **never updated**. Inverse: if local has unsynced edits and remote is older (impossible today ‚Äî no edit flow ‚Äî but the contract is wrong), local wins by accident, not by design. Larger danger: any row deleted locally but not deleted in Supabase comes back as soon as you boot. There is also no `updated_at` column; we have no way to do real conflict resolution. **Fix sketch:** treat Supabase as canonical for synced rows (`synced=true`), local as canonical only for `synced=false`; on pull, replace any local row whose id is in remote AND has `synced=true`, never resurrect ids the user removed. To do this safely we need a local tombstone set in localStorage. Severity: **HIGH-CRITICAL** (data quality erosion + future-resurrection bug waiting).

### C3 ‚Äî `writeLogs` caps at 50; unsynced records silently dropped when over cap

[app.ts:527-529](app.ts#L527). `localStorage.setItem(STORAGE_KEY, JSON.stringify(logs.slice(0, 50)))`. If Allison is offline for a long stretch (international travel, server outage) and lands at 51 sessions, the 51st write evicts the oldest log ‚Äî which may itself be unsynced. Same for hand logs at 200. Today she has 5 sessions, so this is theoretical, but the algorithm is wrong on principle: cap should prefer dropping `synced=true` entries first, then oldest. **Fix sketch:** in `writeLogs`, sort `unsynced first, then synced newest‚Üíoldest`, take first 50; same for hand. Or: refuse to drop unsynced and raise the cap. Severity: **HIGH** (latent ‚Äî will bite once cap is hit).

### C4 ‚Äî `currentExerciseIndex` is incremented BEFORE rest fires; rest screen shows the WRONG exercise context, and on quit-during-rest the wrong index gets persisted

[app.ts:656-685](app.ts#L656). `advanceExercise`: when finishing a main-phase exercise that isn't the last, it does `state.currentExerciseIndex += 1` and THEN calls `startRest()`. Visually fine because the rest screen ignores index, but if anything else reads `getCurrentExercise()` mid-rest (a future feature, an analytics hook, or anybody who keystrokes during rest) it'll see exercise N+1 already. Quit during rest doesn't persist anything anyway, so no immediate data loss, but the sequence is fragile. **Fix sketch:** keep `currentExerciseIndex` pointing at the exercise just finished until rest completes, then advance in the `startRest` onComplete. Severity: **MEDIUM-HIGH** (sets up future bugs; rename/structure fix).

### C5 ‚Äî Round-end and last-main-exercise interaction: rest fires only between exercises, not between rounds ‚Äî but the comparison `state.currentRound < w.rounds` skips a rest when last exercise of round N is the same exercise as first of round N+1

[app.ts:671-680](app.ts#L671). When you hit the last main exercise in round N (N < rounds), the branch increments round, resets index, AND calls `startRest()`. Good. But `state.currentExerciseIndex < list.length - 1` (line 662) catches all non-last cases and ALSO calls `startRest()` ‚Äî meaning round transition AND between-exercise both have rest. That's correct. BUT: there's an off-by-one risk hidden in the `else if` chain: if `currentExerciseIndex === list.length - 1` AND `currentPhase==='main'` AND `currentRound === w.rounds`, we go to cooldown. If `currentExerciseIndex === list.length - 1` AND `currentPhase==='cooldown'`, we fall through to `state.screen = 'post-log'` via the bare `else` ‚Äî but ONLY because the if-chain considers warmup ‚Üí main and main ‚Üí main/cooldown explicitly; the final `else` is "everything cooldown does." So cooldown advancing past its last exercise works ‚Äî by accident, not by intent. **Fix sketch:** make the state transition explicit per phase with a switch ‚Äî see refactor map ¬ß7. Severity: **MEDIUM** (works today, brittle to future edits).

### C6 ‚Äî Race condition: `pullFromSupabase` mutates `localStorage` mid-workout

[app.ts:1501](app.ts#L1501) + [app.ts:849](app.ts#L849). `pullFromSupabase` runs once on DOMContentLoaded, but `flushPendingSyncs` runs after that and `pushLogToSupabase` is called at workout-completion time. If Allison opens the app, immediately starts a workout, and the boot-time pull lands while she's mid-flow, `writeLogs(merged)` ([app.ts:884](app.ts#L884)) rewrites the entire log array ‚Äî fine for her current pre-workout state because there's no unsynced in-flight write. BUT: she saves a session, `pushLogToSupabase` runs, AND if a _new_ boot-pull races (today it can't, single-page; but if you ever add a visibility-refresh listener, it can), the pull would re-fetch what she just pushed and the merge would deduplicate by id correctly ‚Äî UNLESS the just-pushed record hasn't yet returned in the SELECT (read-your-writes lag). Then the next pull dedupes by id `localIds.has(r.id)` ‚Äî local has it, so remote is filtered out. Safe today. **Risk: any future "pull on focus/visibility" feature will need real ordering.** Severity: **MEDIUM** (mitigated today by single-shot pull; document the invariant).

### C7 ‚Äî Background-tab / screen-sleep timer drift

[app.ts:606-619](app.ts#L606). `window.setInterval(..., 1000)` decrements `state.timerSeconds -= 1` per tick. When the tab is backgrounded (phone screen sleeps, Allison switches apps, iOS Safari pauses interval), the interval throttles aggressively (Chromium: 1000 ms minimum; iOS Safari: timers stop entirely when screen locks). So a 25-second wall-sit timer started, screen locked at 5s elapsed, unlocked 60s later, will read "20 seconds remaining" instead of "complete." **Fix sketch:** wall-clock-anchored timer. Store `state.timerEndsAt = Date.now() + seconds*1000`; on each tick compute `Math.max(0, Math.ceil((endsAt - Date.now())/1000))`; add `document.visibilitychange` and `pageshow` listeners that recompute and finish if expired. Severity: **HIGH** (real for Allison ‚Äî she works out at home, phone screen does lock).

### C8 ‚Äî Audio context never resumed on user gesture; iOS Safari + Chrome autoplay policy will silently block beeps on first use

[app.ts:468-498](app.ts#L468). `getAudioCtx()` constructs on first beep, `playBeep` calls `ctx.resume()` if suspended. Per Web Audio autoplay policy (Chrome, Safari, Firefox all enforce it now), `AudioContext` must be resumed from inside a synchronous user-gesture handler. `playBeep` is reached via `startPreCountdown` which runs in a `setInterval` callback ‚Äî not in a gesture. The resume() returns a promise that is `void`-discarded. On first session in a fresh tab Allison hears nothing for the first wall-sit; subsequent beeps work because the context is now running. **Fix sketch:** call `getAudioCtx()` + `ctx.resume()` synchronously on the "Start timer" / "Begin workout" click; verify with `audioCtx.state === 'running'` before playing. Severity: **HIGH** (silent audio is a regression from the May 2 commit `3b58882`).

### C9 ‚Äî Rapid double-tap "Done ¬∑ Next" can fire `advanceExercise` twice and skip a rest

[app.ts:1441](app.ts#L1441). The handler is rebound every render (correct), but the FIRST tap triggers `advanceExercise()` ‚Üí `startRest()` ‚Üí `render()` ‚Üí handler reattached to "Skip rest" button at the same DOM location. If the second tap lands in the ~10-20 ms window where the OLD `Done ¬∑ Next` button hasn't been replaced yet, the second tap hits the now-detached handler closure ‚Äî which re-runs `advanceExercise` against state where `currentExerciseIndex` already advanced. Most likely effect: a second `startRest` call which clears + restarts the rest interval ‚Äî survivable. Worst: phase-skip if she taps as warmup‚Üímain transition is rendering. **Fix sketch:** disable the button at click time or guard `advanceExercise` with a `state.advancing` flag that clears after render. Severity: **MEDIUM-HIGH** (Allison taps fast, has tendinitis, occasional misfires expected).

### C10 ‚Äî `syncDisabled() === navigator.webdriver` can fire in production

[app.ts:104-106](app.ts#L104). `navigator.webdriver` is `true` only under WebDriver-controlled browsers (Playwright, Selenium). BUT: some privacy extensions (Brave shields, hardened Firefox, some Chromium-derived browsers) spoof or set this. If she ever opens the live URL in a privacy browser on her phone, sync is silently off. No user-visible indicator that says "sync is disabled because‚Ä¶" ‚Äî the sync-indicator just shows "offline ¬∑ N pending" forever. **Fix sketch:** keep the guard, but log a console warn when it fires, and add a developer toggle. Or replace with `import.meta.env.MODE === 'test'` ‚Äî but that requires a bundler. Quickest: rename to `isAutomatedTest()` and add a defensive `&& location.hostname === 'localhost'` to limit blast radius. Severity: **MEDIUM** (low probability, awkward debug if it ever fires).

### C11 ‚Äî `formatDate(iso)` and the week banner use `new Date(iso)` without explicit timezone ‚Äî Jerusalem is JDT/IDT, dates near midnight UTC will display wrong

[app.ts:976-979](app.ts#L976), [app.ts:990-998](app.ts#L990). `new Date('2026-05-02T00:00:00')` (no Z) creates a local-time date; `new Date(PROGRAM_START_DATE + 'T00:00:00')` is local ‚Äî fine. But `formatDate(l.date)` receives an ISO string with Z, parses local ‚Äî also fine for display. The week-bucket math uses ms-since-epoch differences divided by 86400000, so DST transitions in non-Jerusalem locales could push a session into the wrong week. Jerusalem doesn't observe DST shifts that affect this (uniform IDT in summer), but the math is naive. **Fix sketch:** convert both `start` and `date` to Jerusalem-midnight before subtracting, OR use a deterministic ISO date string compare for week-bucketing. Severity: **LOW-MEDIUM** (edge case near midnight Jerusalem time).

### C12 ‚Äî `parseInt(wallsitEl.value, 10) || 0` accepts negative / NaN / huge values silently

[app.ts:1459-1460](app.ts#L1459). HTML input has `min="0" max="600"` but those are advisory only ‚Äî JS reads the raw value. Allison enters `-25` ‚Üí `parseInt` returns `-25` ‚Üí stored. Enters `"abc"` ‚Üí `NaN` ‚Üí `|| 0` ‚Üí stored as 0 (correct). Enters `999999` ‚Üí stored. **Fix sketch:** clamp: `Math.max(0, Math.min(600, parseInt(v, 10) || 0))`. Same audit needed everywhere `parseInt` is used (cap-before, cap-after, back; bindRange uses 1-10/0-10 ranges via the slider which clamps natively). Severity: **LOW** (Allison doesn't fuzz her own form, but it's bad hygiene).

### Proposed clean timer model

```
type TimerKind = 'pre-countdown' | 'rest' | 'timed-exercise';
type TimerState = {
  kind: TimerKind;
  endsAt: number;        // Date.now() + duration*1000 ‚Äî wall-clock anchored
  totalSec: number;      // for progress display
  onComplete: () => void;
  lastSecRendered: number;  // throttle render to once per visible second
};

// Single source of truth ‚Äî at most one active timer.
let activeTimer: TimerState | null = null;
let rafHandle: number | null = null;

function startTimerV2(kind: TimerKind, sec: number, onComplete: () => void) {
  stopTimer();
  activeTimer = { kind, endsAt: Date.now() + sec * 1000, totalSec: sec, onComplete, lastSecRendered: sec };
  state.timerSeconds = sec;
  state.timerKind = kind;
  // use requestAnimationFrame loop, gated to ~60ms ‚Äî survives tab throttling
  // because each frame recomputes from endsAt - Date.now()
  loop();
}

function loop() {
  if (!activeTimer) return;
  const remainMs = activeTimer.endsAt - Date.now();
  const remainSec = Math.max(0, Math.ceil(remainMs / 1000));
  if (remainSec !== activeTimer.lastSecRendered) {
    activeTimer.lastSecRendered = remainSec;
    state.timerSeconds = remainSec;
    if (remainSec > 0 && remainSec <= 3) playCountBeep();
    render();
  }
  if (remainMs <= 0) {
    const cb = activeTimer.onComplete;
    activeTimer = null;
    playFinishBeep();
    cb();
    return;
  }
  rafHandle = requestAnimationFrame(loop);
}

function stopTimer() {
  if (rafHandle !== null) cancelAnimationFrame(rafHandle);
  rafHandle = null;
  activeTimer = null;
  state.timerSeconds = 0;
}

// On visibility change, fast-forward to the correct second:
document.addEventListener('visibilitychange', () => {
  if (document.visibilityState === 'visible' && activeTimer) {
    loop();  // immediate recompute; fires onComplete if past endsAt
  }
});

// User-gesture audio unlock ‚Äî call this from the Start button click handler:
function unlockAudio() {
  const ctx = getAudioCtx();
  if (ctx?.state === 'suspended') void ctx.resume();
}
```

Benefits: wall-clock anchored (T2/C7), single source-of-truth replaces the `timerInterval` + closure-local-interval mess (T3/T4), explicit `kind` lets render/audio paths key off the actual timer rather than `isResting`/`preCountdown` toggle ghosts (C1), pause becomes trivial (`endsAt += pauseDuration`), screen-sleep just works.

Recommended fix sequence inside the timer subsystem: **T1 + T7 first (one line, ships the wall-sit fix), then T2/T3/T4/T6 as a single timer rewrite, then T8/T9 as a follow-up data-quality session.**
